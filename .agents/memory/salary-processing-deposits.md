---
name: Salary processing creates categorized deposits
description: How /salary/process turns salary + allocations + loans into real deposit transactions
---

Processing a salary month (`POST /salary/process`) creates REAL deposit transactions, not just budget updates. It is RE-RUNNABLE (idempotent refresh), not a one-shot.

- Requires `salary.accountId`. If null, returns `{processed:false}` with an Arabic message to pick a deposit account first — it does NOT silently skip.
- **Re-runnable**: inside one DB transaction it first takes a per-month advisory lock (`pg_advisory_xact_lock(hashtext('salary-process-'||month))`) to serialize concurrent runs, then DELETEs prior salary deposits for the month (exact note match, see below) and recreates them. The delete is constrained to `type='deposit'` + the exact note shapes so manual deposits/expenses are never touched.
- **Core logic is `runSalaryProcessing(month, {skipDepositDayGuard?})` exported from `routes/salary.ts`**, returning a discriminated union `{kind:"error"|"skip"|"done"}`. The `/salary/process` route is a thin wrapper; the startup backfill reuses the same function so both apply identical rules.
- **Automatic deploy-time backfill** (`lib/backfill.ts`, fired void from `index.ts` before `app.listen`): once per version it finds every month with existing salary deposits (regex `^راتب\s+(\d{4}-\d{2})` over deposit notes) and reprocesses each with `skipDepositDayGuard:true`, so existing/production data picks up new rules (renames, per-loan subs, tagged surplus) WITHOUT the user manually reprocessing. Version-gated by a sentinel row in `salary_processing_log` (`processedMonth='__backfill_..._vN__'`; bump N to re-run). **Sentinel is written ONLY when every month returns `done`** — a skip/error leaves it unset so it retries next boot (never marks complete on partial migration).
- One deposit per allocation. **Category-level allocations (subcategoryId NULL) are NOT deposited as null** — they get a stable default subcategory named after the category via `ensureCategoryDefaultSubcategory` (find-or-create by name), so surplus-style buckets like "الفائض" become visible in the breakdown. Note `راتب {month} - {label}`.
- Remainder is surfaced as a REAL single-sub system category "الإدخار" (sub "إدخار") via `ensureSingleSubSystemCategory`, guarded by a global advisory lock `salary-system-subcategories`.
- **Debt is now PER-LOAN, not shared.** Each active loan gets its OWN subcategory under the "الديون" category, NAMED AFTER THE LOAN (`ensureNamedSubcategory(tx, debtCategoryId, loan.name, "📄")`), so different debts are distinguishable in the breakdown. The legacy shared "الأقساط" subcategory is DELETED after processing — but ONLY when it holds zero transactions (guarded by a count), so any manual data under it is preserved. Once every month is reprocessed, "الأقساط" disappears. (This reverses the earlier "all loans share one الأقساط sub" note.)
- **Renames are applied IN PLACE.** Helpers take `aliases` + canonical names; they find the category by ANY alias (e.g. legacy "المتبقي من الراتب") and UPDATE it to the canonical name/emoji, and rename the single existing subcategory in place. Because the subcategory ID is preserved, historical deposits keep showing under the new name.
- **The DELETE predicate matches EXACT system note shapes**, not a loose prefix: `notes = 'راتب {month}'` OR `notes LIKE 'راتب {month} - %'` (uses drizzle `or`). This prevents an unrelated manual note that merely starts with "راتب {month}" from being caught. If you change the note format, update BOTH the created notes and this predicate in lockstep.
- One deposit per ACTIVE loan (`loansTable` where `isActive`), amount = `monthlyInstallment`, subcategoryId = debt sub, note `راتب {month} - قسط {loan.name}`.
- Final remainder deposit = `salary - sum(allocations) - sum(active loan installments)`, subcategoryId = remainder sub, note `راتب {month} - غير موزع` (or full `راتب {month}` when nothing committed).
- **Why real subcategories:** the account-breakdown (`summary.ts`) groups by subcategoryId. Any NULL-subcategory money is now bucketed into a synthetic "غير مصنف" category (id -1) so it is never hidden and `sum(category nets) + opening balance == current balance` ALWAYS holds. Tagging remainder/debt/surplus makes the per-category totals meaningful AND makes them transferable/spendable. Legacy null-subcategory salary rows from older months still only get their real tag when that month is reprocessed (until then they show under "غير مصنف").
- Guard: reject (`processed:false`) if `allocations + loan installments > salary`. Processing-log insert uses `.onConflictDoNothing()` (log is informational only, no longer blocks).

**Why:** User's mental model is that the FULL salary is deposited into the account and split into buckets — categories AND debt/loans — so they can then transfer portions anywhere. A stale bug (old code logged a month as processed even when accountId was null, creating no deposit) permanently blocked re-processing via an "already processed" early-return; that early-return was removed so processing always deposits. Keep total deposited == salary.amount in every branch.

**How to apply:** Never re-add an "already processed" early-return — the delete-and-recreate flow is what makes it self-healing. The DELETE match relies on the `راتب {month}` note prefix; if you change the note format, update the delete predicate in lockstep or reprocessing will leave duplicates. Loans ARE deposited now (as debt) — this reverses the earlier "loans are not auto-converted" note.
